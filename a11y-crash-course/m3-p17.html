A table that explains the relationship between different types of information is a data table.

Assistive technologies use header information to identify rows and columns. Always specify row and column headers.

Accessible tables are:

  • simple, rather than complex,
  • have an identified header row, and
  • include a table summary, either as a caption or as alt text.

These techniques help people with assistive technology read the information contained in the table.

When sighted users focus on a table cell, they are able to visually determine which row and column the cell is in by scanning up and down or left and right. This adds context to what the value in a particular cell means.

Did you know? A screen reader can only read aloud each table cell one by one from left to right and top to bottom. If the table is not formatted correctly, determining what label a particular value in a cell might have is not quick or easy.

Example: If you are not a screen reader user, pretend that you are and you want to find the location of a biology 205 class. You find a webpage with an incorrectly formatted table and you hear the following text read out loud. Do you know where the class will be held?

Creating Accessible Tables from WebAIM

Tables checklist

  • Use tables to display data, not to format content.
  • Designate at least one row or column header.
  • Don’t merge cells.
  • Simplify complex tables or break them down into multiple tables.
  • Provide contextual information like captions, descriptions, or HTML data to describe the table.
  • Include content in all cells of the table. For cells that don’t have any content, use “not applicable” or “none.”

Details

Use tables to display relational information

Tables traditionally show the relationship between two or more items in rows and columns. But in the early days of the internet, web designers and developers commonly used tables to control the layout of webpage content as well because they couldn't easily do so with HTML.

Similarly, you may have used tables in word processing documents to control where content is placed rather than to indicate data relationships. But screen reader software may not read cells in such tables in the proper order, and page layout can now be better controlled with CSS on webpages and with other tools in document creation software.

Key point: Don't use tables to make your webpage or document look a particular way. Tables do not pose inherent accessibility issues, but it is more difficult to make sure screen reader software reads the cells in the proper order (10 minute read).

Designate row and/or column headers

When a table has appropriate row and column headers, blind or visually impaired people can process the information displayed in a table, as their their screen reader software reads the row and column headers before reading the content in each cell.

  • Designate at least one row and/or column header using the table formatting tools in your web content management system or document creation software.

    Technical note: If you are using HTML, use the <th> element to mark up table headers. Then identify the header's scope for row or column using the <scope> attribute.

  • Don't just change the visual formatting of the text, such as the font size or color, to indicate table header rows and/or columns. Screen readers will not be able to associate the headers with the correct cells.

Important: Table headers should never be empty. This is particularly of concern for the top-left cell of some tables.

Avoid or simplify complex tables

Simple tables are easier for people using assistive technologies to understand and navigate, as the data structure is so simple that screen readers can easily navigate it. HTML code maintenance may be easier for simple tables as well.

  • Include a maximum of one header row and one header column.
  • Don't merge cells, and don't include a table within another table. If your table has multiple header rows, merged cells, or another table embedded in it, split it into two or more simple tables.
  • Spell out abbreviations or acronyms

Technical note: If using HTML, use the <abbr> or <acronym> tags to ensure accessibility.

Important: If you do create a complex data table on a webpage using HTML, use the <scope> tag to programmatically associate the data cells with the appropriate headers.

Provide contextual information

Data tables often have contextual information around them that helps users understand the content of the table.

  • Associate descriptive text about a table with its respective table by including a <caption> element in HTML or alt text in Microsoft Word. Captions are not required for each table but can be helpful for screen reader users. The caption can be visually formatted and positioned above or below the table as needed, but on webpages, the <caption> element must be the first one after the opening <table> tag.
  • Don't repeat the same text in the caption that appears in a heading preceding the table.
  • Don't leave any table cells empty.
  • When using HTML, you may use <thead>, <tfoot>, and <tbody> tags in HTML tables so that the head and/or foot rows repeat at the top or bottom of the table when it is printed, but these do not provide any additional accessibility benefits.
  • If using HTML, you may provide a summary of the structure of the data table (not of the content) using the <summary> attribute, but screen reader support for it varies, and it is not part of the HTML5 specification, so WebAim does not recommend it. If both a caption and summary are provided for one table, the summary should not duplicate information present in the caption.
