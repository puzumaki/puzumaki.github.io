Interactions

Objective: Ensure the entire site can be operated without a mouse.

Tools Needed: JAWS or NVDA (Windows), VoiceOver (Mac).

Websites often use more complex components to selectively reveal content, such as clicking a login button to open a login dialog box.

These more complex components have similar navigational challenges as forms: they need to be clear about status, proper labels, and include instructions. If errors occur, the person needs to be able to easily understand and correct the errors.

Forms

Forms

Forms allow people to enter information, but it can be impossible for people using assistive technology if it isn't programmed correctly.

  • Form fields can be heard, accessed by a keyboard, and interacted with as expected: Assessment 2.4 On input
  • Each field is using the correct field type: Assessment 8.1 Widget function
  • Form fields have visible labels or instructions that are adjacent and programmatically connected to the field: Assessment 8.2 Instructions
  • The form labels identify the expected input using <label>, aria-label, or aria-labelledby: Assessment 8.3 Expected input
  • Fields provide appropriate cues for its state including disabled, read-only, and required: Assessment 8.4 Cues
  • Any fields that can be autocompleted are coded to support autocomplete: Assessment 8.5 Autocomplete

How to test

Desktop screen reader

Desktop screen reader

Given that I am on a page with form fields:

  • When I use a desktop screen reader (NVDA, JAWS, VoiceOver), and
  • I use the arrow keys to browse the navigation menu or navigate through each form field:
    • I hear all form fields have descriptive labels that describe the purpose of the field, and
    • I hear all radio buttons and checkboxes are appropriately grouped with descriptive group labels, and
    • I hear the correct status conveyed when I change the status of a form field, tab, or other control (e.g. "selected", "not selected", "checked", "not checked", "expanded", "collapsed"), and
    • I see and hear that the visual label matches the accessible (spoken) label.

Error identification

Error identification

A person must be able to navigate from an error message to the field that needs repair and understand what changes need to be made.

  • An automatically detected error must be identified and described in text: Assessment: 11.1 Error identification
  • Guidance to correct an error must be provided: Assessment 11.2 Error suggestion
  • If submitting data might have serious consequences, allow for review and edits before submitting or an undo: Assessment 11.3 Error prevention
  • Status messages are announced, including successful actions, waiting state, progress update, or errors:Assessment 11.4 Status messages

How to test

Desktop screen reader

Desktop screen reader

Given that I am on a page where user input errors are possible:

  • When I make a user input error:
    • I hear that the error is programmatically associated with the correct form field, and
    • I hear an alert or the screen reader focus automatically jumps to information about the errors on the page.
    • I see (or hear if using a screen reader) information on what the error is and how I can fix it, and.
    • Where possible, I see (or hear if using a screen reader) instructions that will help prevent user input errors (e.g. expected formats, required fields), and
    • If the form includes financial, legal, or deletion or modification of user data in the data storage system:
      • I see (or hear if using a screen reader) a mechanism to reverse, check and correct, or confirm the submission.

Dynamic content

Dynamic content

Because the content is dynamic, the page must announce the changes not only visually, but it's coded correctly so it can announce it to assistive technology. It must respond predictably, nothing is unexpected. And the changes should not be substantial enough to cause confusion or disorientation unless the person is informed of the change in advance.

  • Revealing hidden content must move the focus into the revealed content: Assessment 3.2 Revealing content
  • For modals, tabbing must be contained to the modal dialog until the modal is explicitly dismissed: Assessment 3.3 Modal dialogs
  • Closing revealed content must return the focus to the component that revealed it: Assessment 3.4 Closing content
  • Custom components have appropriate ARIA roles and their states and properties, communicating those states programmatically, not only change visually: Assessment 9.1 Design pattern Assessment 9.4 Role, state, property Assessment 9.5 Cues
  • A custom component with visible label or instructions that clearly state expected input are programmatically tied to the component: Assessment 9.2 Instructions Assessment 9.3 Expected input
  • A custom component must support keyboard interactions: Assessment 9.6 Keyboard interaction

How to test

Desktop screen reader

Desktop screen reader

  • When I come across a custom interactive component,
    • I can activate it with the keyboard
    • the focus moves to the new content
    • if it opens content, the content can be closed with a keyboard
    • I can hear the changes and different states announced as I use the component.
