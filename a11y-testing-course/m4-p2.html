Accessibility Testing Course

Page structure

Objective: Verify that the underlying programmatic structure matches the visual presentation for blind or low-vision users.

Tools needed: JAWS or NVDA (Windows), VoiceOver (Mac).

Page title

Page title

Make sure a <title> element is present in the <head> and contains text. Titles should be concise, unique, accurate, informative, and similar to the content in the first heading.

  • Page title describes the page's topic or purpose: Assessment: 12.1 Page title

How to test

Desktop screen reader

Desktop screen reader

  • When I hover over the browser tab OR open the page with a screen reader:
    • I see or hear a descriptive and unique page title.

Landmarks

Landmarks

All text should be within a landmark region, and the number of landmarks should be minimal. A page should only have one <header> (or role="banner"), one <main>, and one <footer> (or role="contentinfo"). If multiple of one type of landmark such as <nav> exists, use aria-label or aria-labelledby to differentiate them.

  • No repeating content should be in the one <main> element, only the page's primary content: Assessment: 4.2 Primary content, Assessment 4.3 No repeating content
  • Content is in the most appropriate landmark region Assessment 4.1 Landmark roles.

How to test

Desktop screen reader

Desktop screen reader

  • When I use a desktop screen reader (NVDA, JAWS, VoiceOver), and
  • I use the arrow keys to browse the navigation menu:
    • I hear it indicates its role, and if multiple navigations are present (ex: Main navigation, Site map, Breadcrumbs), the name of the navigation.
    • I hear it is discoverable with screen reader shortcuts as a navigation landmark.
Mobile screen reader

Mobile screen reader

  • When I use a mobile screen reader (Talkback, VoiceOver), and
  • I swipe to elements in the navigation:
    • I hear it indicates its role, and if multiple navigations are present (ex: Main navigation, Site map, Breadcrumbs), the name of the navigation.
    • I hear it is discoverable with screen reader shortcuts as a navigation landmark.

Headings

Headings

Headings should provide a clear and accurate outline of the content structure of a page. Headings should be informative, concise, meaningful, and properly nested. They can be a good way to quickly navigate a page.

  • Headings are properly coded as headings (not only styled to look like one): Assessment 5.1 Heading function Assessment 5.2 No missing headings
  • The heading levels are styled to visually match their programmatic level (a heading level 1 is the most prominent and level 6 is least): Assessment 5.3 Heading level
  • Headings are not focusable unless the text is a link or the heading is used to open or close an accordion.

How to test

Desktop screen reader

Desktop screen reader

  • When I use a desktop screen reader (NVDA, JAWS, VoiceOver), and
  • I use the arrow keys to browse to a heading. Alternatively, use "H" key on a keyboard while on Windows with a desktop screen reader. For VoiceOver, you can select headings from the rotor selection using ctrl + command + option and right arrow.
    • I hear the heading's purpose and level must be clear.
    • I hear it identifies itself as a heading and its level.
    • I hear it is logically ordered, starting with a single Heading level 1, sections titled by a Heading level 2, and subsections with a Heading level 3.
  • When I use the Tab key, I hear nothing happens to the headings because headings must NOT be focusable.
Mobile screen reader

Mobile screen reader

  • When I use a mobile screen reader (Talkback, VoiceOver), and
  • I swipe to browse to a heading:
    • I hear the heading's purpose and level must be clear.
    • I hear it identifies itself as a heading and its level.
    • I hear it is logically ordered, starting with a single Heading level 1, sections titled by a Heading level 2, and subsections with Heading level 3.

Tables

Tables

Tables are primarily used for relational content (e.g. number of students in classes over 3 semesters, where the headers would be the class name and the semester).

  • Content in a table without relationships is purely visual and needs to be designated as such with <table role="presentation"> or <table role="none">: Assessment 22.2 Semantics
  • Data table headers are properly coded (either a <th> with its scope like <th scope="row"> or giving a <td> the role with <td role="rowheader">) and accurately describe the content: Assessment 22.3 Table headers
  • For a data table with complex headers, each table cell needs a header attribute to tie it to the right headers, and each <th> needs an ID for the header to associate (e.g. <th id="title"> and <td header="title">): Assessment 22.4 Headers attribute

How to test

Desktop screen reader

Desktop screen reader

When using a screen reader to navigate a table:

  • the screen reader speaks one cell at a time and explains where each cell is in the table. Tip: When using JAWS, press the ctrl + alt keys to navigate within the table.
  • you hear the table cells for an entire row read in the correct order (e.g., from left to right for English) before moving down to the next row.
  • you can clearly hear which column or row header the data falls under.
  • with merged header cells, you can understand which lower-level data is organized under merged header cells.
  • the screen reader announces the name and role of interactive elements such as scrolling, sorting, and clicking. For sortable tables, you hear whether the column is sorted and how it is sorted.
